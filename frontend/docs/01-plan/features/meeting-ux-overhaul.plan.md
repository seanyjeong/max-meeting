# Plan: Meeting UX Overhaul

> 회의 화면 UX 전면 개편 - 탭 구조 개선, 화면 크기 확대, 메모/스케치 결과 표시

## 1. 배경 및 목적

### 현재 문제점
1. **화면 크기**: 태블릿(2000x1200)에서 너무 좁음, 안건 영역 특히 협소
2. **탭 이름 혼란**: "텍스트", "스케치", "실행" - 직관적이지 않음
3. **업무배치 기능 부재**: 회의 중 담당자 배정 탭 없음
4. **메모/스케치 미표시**: 회의록 결과에서 메모 내용과 스케치가 보이지 않음

### 목표
- 태블릿 화면을 최대 1900px까지 활용
- 직관적인 탭 명칭으로 변경
- 회의 중 업무배치 기능 추가
- 회의록에 메모를 포스트잇 스타일로 표시

## 2. 변경 범위

### 2.1 탭 명칭 변경 (회의 중 화면)
| 현재 | 변경 후 | 설명 |
|------|---------|------|
| 텍스트 | 메모 | 텍스트 메모 입력 |
| 스케치 | 필기 | 펜/그림 입력 |
| (없음) | 업무배치 | 담당자 배정 (NEW) |
| 실행 | (삭제) | 업무배치로 통합 |

### 2.2 화면 크기 확대
- 현재: max-width 제한 (약 1200px 추정)
- 변경: 최대 1900px, 반응형 유지
- 영향 범위: 전체 레이아웃 (`+layout.svelte`, 주요 페이지)

### 2.3 업무배치 탭 (신규)
- 회의 중 담당자 배정 UI
- 결과 페이지의 "실행항목"과 데이터 연동
- 안건별로 담당자/마감일 배정 가능

### 2.4 회의록 메모 표시 (결과 페이지)
- 메모 내용을 포스트잇 스타일로 표시
- 해당 안건 옆에 포스트잇 형태로 배치
- 스케치는 별도 갤러리 또는 링크로 접근

## 3. 상세 요구사항

### 3.1 화면 크기 (Priority: HIGH)
```
Target Device: 2000x1200 태블릿
Max Content Width: 1900px
Breakpoints:
  - mobile: < 768px
  - tablet: 768px - 1200px
  - desktop: 1200px - 1600px
  - wide: 1600px - 1900px
```

### 3.2 탭 구조 (Priority: HIGH)
```
회의 중 화면 탭:
┌──────────┬──────────┬──────────┐
│   메모   │   필기   │  업무배치 │
└──────────┴──────────┴──────────┘
```

### 3.3 업무배치 탭 UI (Priority: MEDIUM)
```
┌─────────────────────────────────────────┐
│ 업무 배치                               │

│ [+ 업무 추가]                           │

│ └─────────────────────────────────────┘ │
│ ┌─────────────────────────────────────┐ │

│ │   담당: -     마감: -               │ │
│ └─────────────────────────────────────┘ │
└─────────────────────────────────────────┘
```

### 3.4 포스트잇 메모 표시 (Priority: MEDIUM)
```
결과 페이지 안건 옆:
┌─────────────────────────────────────────────────┐
│ 1. 예산 검토                                    │
│    - 요약 내용...                               │
│                        ┌──────────────────────┐ │
│                        │ 📝 메모              │ │
│                        │ "예산 10% 삭감 검토" │ │
│                        │ "김OO 의견 반영 필요"│ │

└─────────────────────────────────────────────────┘
```

### 3.5 스케치 접근 (Priority: LOW)
- 옵션 A: 결과 페이지 하단에 "스케치 보기" 버튼
- 옵션 B: 안건별 스케치 썸네일 표시
- 옵션 C: 별도 "스케치 갤러리" 페이지

## 4. 기술 고려사항

### 4.1 데이터 구조
- 메모: 현재 `notes` 필드 활용 (안건별 저장 필요 확인)
- 스케치: `tldraw` 데이터 저장 위치 확인
- 업무배치: 기존 `action_items` 테이블 활용

### 4.2 영향받는 파일 (예상)
```
Frontend:
- src/app.css (전역 width)
- src/routes/+layout.svelte (컨테이너 width)
- src/routes/meetings/[id]/+page.svelte (회의 중 탭)
- src/lib/components/recording/*.svelte (탭 컴포넌트)
- src/routes/meetings/[id]/results/+page.svelte (결과 페이지)
- src/lib/components/results/*.svelte (결과 컴포넌트)
```

### 4.3 API 변경
- 메모 저장/조회 API 확인 필요
- 스케치 데이터 저장 구조 확인 필요

## 5. 작업 분류

### Phase 1: 기반 작업 (화면 크기 + 탭 명칭)
1. 전역 max-width 1900px 확대
2. 탭 명칭 변경 (텍스트→메모, 스케치→필기)
3. "실행" 탭 제거

### Phase 2: 업무배치 탭
1. 업무배치 탭 UI 구현
2. action_items와 연동
3. 회의 중 ↔ 결과 페이지 동기화

### Phase 3: 메모 표시
1. 안건별 메모 데이터 구조 확인
2. 포스트잇 스타일 컴포넌트 제작
3. 결과 페이지에 메모 표시

### Phase 4: 스케치 접근
1. 스케치 데이터 저장 구조 확인
2. 접근 방식 결정 (브레인스토밍 필요)
3. 구현

## 6. 미결정 사항

| 항목 | 옵션 | 결정 필요 |
|------|------|-----------|
| 스케치 접근 방식 | 갤러리 / 썸네일 / 버튼 | 브레인스토밍 후 결정 |
| 메모 저장 단위 | 회의 전체 / 안건별 | 현재 구조 확인 후 결정 |
| 포스트잇 위치 | 안건 옆 / 안건 하단 | 디자인 단계에서 결정 |
